| 항목 | 내용 |
|------|------|

## 2. 전체 일치도 요약

| 영역 | 판정 | 핵심 차이 한 줄 요약 |
|------|------|---------------------|
| 전반 레이아웃 (사이드바/헤더/메인/우측 또는 하단 패널) | 일치 | 우측보기/하단보기 토글까지 동일 |
| 사이드바 메뉴 구성 | 변경 | mock 4개 vs 외부 10개 (대시보드, 점검대상, 점검이력, 제외신청관리, 조치계획관리, 커뮤니티, 정책관리, 대상관리, 보안관리 등 추가) |
| 헤더(Hero) | 부분일치 | "검출목록" 타이틀은 동일하나 mock의 `[target명]` 부제와 **사용자 메뉴(사용자설정/로그아웃 dropdown)가 헤더에 없음** |
| 검출목록 툴바 | 부분일치 | 검색이 **카테고리(검출경로/DB명/스케줄ID) + 입력값**으로 분리, "선택 해제" 버튼 추가 |
| 상세필터 | 일치 | 검출타입/담당자/이슈상태 3종, 적용/초기화 동일 |
| 테이블 컬럼 (9개) | 일치 | 체크박스 / 검출ID / 스케줄ID / DB명 / 검출경로 / 검출타입 / 건수 / 담당자 / 상태 — 순서·종류 일치 |
| 정렬 | 변경 | URL `?sortKey&sortDir` 라우팅으로 동작 (mock은 client state) |
| 페이지 사이즈 | 추가 | 5/10/20 + **50/100 추가** |
| 페이지네이션 | 일치 | 1, 2, ... 14 형식 동일 |
| 행 선택/일괄 액션 | 일치 | 수정/삭제/조치계획/내보내기/이행점검 5종 동일 |
| 일괄수정 모달 | 변경 | 상태 변경이 **버튼 그리드가 아니라 dropdown(상태 선택)** 으로 구현 |
| 우측 상세 패널 | 변경 (구조 분리) | mock 3개 카드 vs 외부 **4개 카드(개인정보/검출내용/상세값/검출결과 상태)** |
| 검출결과 상태 카드 | 부분일치 | 상태 옵션이 **`조치완료` 누락**(미확인/조치필요/제외신청/제외/제외거부 5개) |
| PII(개인정보) 검색 | 부분일치 | 외부는 placeholder 노출 없음, 검색창만 표시. 정렬/페이징은 동일 |
| 상세 정보 카드 | 부분일치 | mock의 "Unique 정보 + 컨텍스트 ul" → **검출 값/검출 건수/유니크키/조회 쿼리 4개 필드**로 분해, "검출내용"은 별도 페이징 테이블로 분리 |
| 변경 이력 모달 | 변경 | 4컬럼(변경일시/변경자/변경상태/의견) — **mock의 `담당자` 컬럼 누락**, 헤더 표시명 "변경이력" |
| 역할 기반 권한 | 누락 | mock의 관리자/일반사용자 role-switch 위젯 없음 (사이드바 하단의 "Admin User · Global Admin" 표시만) |
| URL 라우팅 파라미터 | 변경 | mock(`dbId/target/detectType/detectionId/detectionQuery/piiQuery`) → 외부(`page/pageSize/sortKey/sortDir/panelId`) |
| 시각 디자인 | 일치 | 보라색 컬러, 라운드 카드, 칩(chip) 스타일 모두 mock 가이드와 동일 |

### 9.1 보고서 1차 분석 시 미관찰 → 2차 추가 발견

- **검출결과 상태 카드의 `마지막 변경`은 저장 즉시 갱신**된다. mock은 `상태변경일시` 텍스트 라벨을 사용하지만 외부는 `마지막 변경 YYYY-MM-DD HH:MM` 형태로 즉시 노출.
- 변경이력 모달의 **`변경자`는 로그인 ID가 아닌 사용자 표시명**(`Admin User`)이 들어간다. mock은 `getCurrentActor()`에서 역할명(`관리자`/`일반사용자`)을 사용 — 외부는 사용자 프로필명을 사용하므로 추적 정밀도가 더 높음.
- 단건 상세 패널의 "**주변 컨텍스트**" 영역(검출 값을 노란 배경으로 강조)이 **상세값 카드 하단**에 별도 블록으로 존재한다. 1차에서는 일부 데이터에만 보여 누락 가능성으로 적었으나, 데이터에 따라 노출되는 정보임이 확인됨 → mock의 `<ul>` 컨텍스트 표현과 가장 가까운 부분.
- **이행점검은 비동기**(요청 → 즉시 시작 토스트 → 백그라운드 처리). mock은 `service.recheckDetections()`로 동기 처리하므로 동작 모델 자체가 다름. 신뢰도 표 작성 시 "외부는 작업 완료까지 추가 갱신/폴링이 필요"로 표기 권장.

### 9.2 권장 추가 조치 (우선순위)

- 🔴 단건 저장 시 **토스트 메시지 일관 표시** (현재 약함). mock 수준의 `검출결과 상태 정보가 저장되었습니다.`처럼 명시적 피드백 필요.

- 🟡 토스트 색상 가이드 정리 — 외부는 success/info만 사용하는 듯 (삭제도 success 색). mock 가이드의 danger/success/default 분리에 맞추면 좋음.
