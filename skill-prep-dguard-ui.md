# D-Guard UI Skill Prep

## 목적
- D-Guard 웹 솔루션의 여러 화면에 공통 디자인과 상호작용 규칙을 일관되게 적용하기 위한 Skill 작성 준비 문서
- 현재 구현된 [index.html](F:\dev\workspace\dguard-ui\index.html), [app.js](F:\dev\workspace\dguard-ui\app.js), [styles.css](F:\dev\workspace\dguard-ui\styles.css)와 기존 요구사항을 기준으로 공통 요소를 추출한다

## Skill에 넣을 공통 범위

### 1. 레이아웃 규칙
- 좌측 사이드바 + 우측 본문 구조를 기본 레이아웃으로 사용한다
- 사이드바는 펼침 `200px`, 접힘 `64px` 규칙을 사용한다
- 상단 헤더는 단일 카드형으로 유지한다
- 본문은 흰색 카드 단위로 분리하고, 카드 간 간격은 촘촘하게 유지한다
- FHD, 크롬 100% 기준에서 주요 정보가 한 화면에 들어오도록 밀도를 높인다

### 2. 디자인 토큰
- 페이지 배경: `#EDF0F5`
- 카드 배경: `#FFFFFF`
- 메인 포인트: `#987BE9`
- 주요 버튼: `#727CF4`
- 버튼 글자색: `#FFFFFF`
- 표는 세로줄 없이 가로 구분선과 여백으로만 구성한다
- 전체 톤은 운영형 엔터프라이즈 UI로 유지한다

### 3. 카드와 섹션 패턴
- 모든 주요 콘텐츠는 라운드 카드 안에 배치한다
- 카드 헤더, 본문, 툴바, 페이지네이션 영역을 명확히 분리한다
- 빈 상태, 비활성 상태, 선택 상태를 시각적으로 구분한다

### 4. 상단 검색/액션 바 패턴
- 기본 구성은 `검색 input + 상세필터 버튼 + 필터 상태 표시 + 액션 버튼들`
- 한 줄 배치를 기본으로 한다
- 검색 input은 너무 짧지 않게 유지하고, 액션 버튼은 우측 정렬한다
- 현재 구현 기준 검색 input 폭은 `280px`이다
- 현재 구현 기준 주요 액션 버튼은 동일 폭을 유지한다
- 필터 적용 중이면 배지와 초기화 액션을 노출한다

### 4-1. 헤더 패턴
- 헤더는 `페이지 제목 + 강조 대상명 + breadcrumb + 사용자 메뉴` 구조를 기본으로 한다
- 페이지 제목은 좌측에 크게 표시한다
- 대상 DB나 시스템명은 제목 오른쪽에서 pill 형태로 강조할 수 있다
- breadcrumb는 항상 표시하고 구분자는 `>`를 사용한다
- breadcrumb 항목은 클릭 가능한 링크처럼 보여야 하며 hover 상태를 제공한다
- 헤더 우측에는 사용자 메뉴를 둔다
- 사용자 메뉴는 `아이콘 + 사용자명(식별자) + 드롭다운 표시` 형태를 기본으로 한다

### 5. 상세필터 패턴
- 상세필터는 전체 모달이 아니라 버튼 아래 popover를 사용한다
- 내부 필터 항목은 검색 가능한 멀티선택 드롭다운으로 제공한다
- 선택값은 칩 형태로 표시한다
- 칩의 `×` 버튼으로 개별 해제 가능해야 한다
- 체크 즉시 반영하지 않고 `적용 / 취소` 버튼으로 반영한다
- 기본 placeholder는 `전체`를 사용한다
- 검색 결과가 없을 때 `검색 결과 없음`을 표시한다

### 6. 테이블 패턴
- 헤더는 배경색을 분리해서 본문과 구분한다
- 세로줄 없이 구성한다
- row hover, row selected 상태를 명확히 제공한다
- 선택 row는 배경색과 좌측 포인트 바로 표현한다
- row click과 checkbox click은 분리한다
- 정렬 가능한 컬럼은 헤더 버튼과 방향 표시(`▲`, `▼`)를 사용한다
- 숫자 컬럼은 기본 우측 정렬, 필요한 예외 컬럼은 중앙 정렬 클래스를 별도로 둔다
- 많은 데이터는 페이지네이션으로 처리한다

### 7. 페이지네이션 패턴
- 하단에 `1-5 / 20건` 형식의 캡션을 둔다
- `이전 / 페이지 번호 / 다음` 구조를 기본으로 한다
- 필터와 정렬 결과를 반영한 상태에서 동작해야 한다

### 8. 상태 변경 패턴
- 상태 버튼은 그리드 형태로 배치한다
- textarea + 글자 수 표시를 함께 제공한다
- 담당자는 검색 가능한 멀티선택 드롭다운으로 제공한다
- 저장 버튼은 변경사항이 있을 때만 활성화한다
- 권한에 따라 상태 선택 범위와 버튼 노출을 달리한다

### 9. 피드백 패턴
- 저장, 삭제, 복사, 이행점검 결과는 toast로 알린다
- 삭제, 일괄수정 등은 모달을 사용한다
- empty state 문구를 각 영역에 제공한다
- disabled 상태는 시각적으로 명확해야 한다

### 10. 권한 시연 패턴
- 관리자 / 일반사용자 전환 UI를 제공한다
- 관리자 전용 버튼과 일반사용자 제한 상태를 분리한다
- 테스트용 권한 전환은 사이드바 하단에 둔다

### 11. 코드 구조 원칙
- UI 상태/이벤트/렌더링은 `app.js`
- 목데이터/샘플 비즈니스 로직은 `mock-service.js`
- 공통 Skill에서도 이 구조를 기본 권장 패턴으로 명시한다

## Skill 작성 전 체크리스트
- 한국어 라벨이 깨지지 않는 기준본을 유지한다
- 디자인 토큰 이름을 고정한다
- 버튼 높이, 폰트 크기, radius, spacing 기준을 숫자로 확정한다
- 필터/정렬/페이징 패턴을 모든 페이지에 동일 적용할지 합의한다
- 페이지 전용 규칙과 공통 규칙을 혼합하지 않는다
