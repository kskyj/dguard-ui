# D-Guard UI Skill Prep

## 목적
- D-Guard 웹 UI를 여러 페이지로 확장할 때 공통 디자인, 상호작용, 밀도 기준을 일관되게 적용하기 위한 Skill 작성 준비 문서다.
- 기준 소스는 현재 구현된 [index.html](F:\dev\workspace\dguard-ui\index.html), [styles.css](F:\dev\workspace\dguard-ui\styles.css), [app.js](F:\dev\workspace\dguard-ui\app.js), [mock-service.js](F:\dev\workspace\dguard-ui\mock-service.js)와 초기 요구사항/계획 프롬프트다.
- 이 문서는 `skill-creator`로 Skill을 만들기 전, Skill 본문에 넣을 공통 요소와 페이지별로 따로 입력받을 요소를 구분하는 용도다.

## 1. Skill로 고정할 공통 범위

### 1.1 레이아웃 규칙
- 기본 레이아웃은 `좌측 사이드바 + 우측 메인 화면` 구조를 사용한다.
- 사이드바는 펼침 `200px`, 접힘 `64px`를 기본값으로 사용한다.
- 메인 화면은 `상단 헤더 카드 + 본문 카드 영역` 구조를 사용한다.
- 본문은 카드 단위로 나누고, 카드 간 간격은 촘촘한 운영형 UI 밀도를 유지한다.
- FHD, 브라우저 100% 기준에서 주요 정보가 한 화면에 최대한 들어오도록 설계한다.

### 1.2 헤더 패턴
- 헤더는 항상 `페이지 제목 + 강조 대상명 + breadcrumb + 사용자 메뉴` 구조를 사용한다.
- 헤더는 현재 샘플처럼 `compact white card` 스타일을 유지한다.
- 헤더는 불필요하게 높거나 여백이 큰 hero 배너 형태로 바꾸지 않는다.
- 페이지 제목은 좌측에 크게 배치한다.
- 강조 대상명은 제목 오른쪽에 pill 형태로 배치한다.
- breadcrumb는 제목 아래에 항상 노출한다.
- breadcrumb 구분자는 `/`가 아니라 `>`를 사용한다.
- breadcrumb 항목은 클릭 가능한 링크처럼 보여야 하며, hover 시 색상 변화와 underline을 제공한다.
- 현재 페이지 breadcrumb는 hover 없이 더 진한 텍스트로 표시한다.
- 헤더 우측에는 사용자 메뉴를 둔다.
- 사용자 메뉴는 `아이콘 + 사용자명(사번) + 드롭다운 표시` 형태를 기본으로 한다.

### 1.3 사이드바 패턴
- 사이드바는 카드형 흰 배경이 아니라 보라색 계열 패널을 기본으로 사용한다.
- 메뉴는 아이콘 + 라벨 구조를 사용한다.
- 접힘 상태에서는 아이콘만 보이고 라벨은 숨긴다.
- 테스트용 전환 UI가 필요한 경우 사이드바 하단에 배치한다.

## 2. 디자인 토큰
- 페이지 배경: `#EDF0F5`
- 카드 배경: `#FFFFFF`
- 메인 포인트 색상: `#987BE9`
- 주요 버튼 색상: `#727CF4`
- 버튼 텍스트 색상: `#FFFFFF`
- 테이블은 세로줄 없이 가로 구분선과 여백으로 구분한다.
- 전체 톤은 장식적이기보다 운영형 엔터프라이즈 UI에 가깝게 유지한다.

## 3. 타이포/밀도 기준

### 3.1 헤더
- 페이지 제목: `18px`, `font-weight: 800`
- 강조 대상명 pill: 현재 구현 기준 강조형 라벨로 사용
- breadcrumb: `12px`
- breadcrumb 현재 항목: `font-weight: 700`
- 사이드바 라벨: `14px`, `font-weight: 700`
- 사용자 메뉴 trigger는 compact pill 스타일을 유지한다.

### 3.2 테이블
- 검출목록 본문 셀: `14px`
- 검출목록 헤더 셀: `13px`
- 개인정보 목록 본문 셀: `13px`
- 개인정보 목록 헤더 셀: `12px`
- 새 페이지를 만들 때 이 크기보다 임의로 더 작게 줄이지 않는다.
- 숫자 컬럼은 기본 우측 정렬을 사용한다.
- 식별자 컬럼처럼 필요한 경우만 중앙 정렬을 별도로 허용한다.

### 3.3 입력/버튼
- 일반 버튼 기본 높이: `32px`
- 일반 버튼 기본 글자 크기: `11px`
- 주요 액션 버튼 최소 폭: `92px`
- 현재 샘플에서 사용하는 주요 액션 버튼은 이모지 + 텍스트 조합을 기본값으로 유지한다.
- 예: `🔎 상세필터`, `✏️ 수정`, `🗑 삭제`, `🔄 이행점검`, `📥 내보내기`
- 상세필터 팝오버 내부 `적용/취소` 버튼 높이: `36px`
- 상태 버튼 높이 기준: `24px` 전후 밀도 유지
- 검색 input 기본 폭: `280px`
- 상태변경 textarea 입력 폰트: `13px`
- 상태변경 textarea placeholder 폰트: `12px`
- 글자수 카운트 폰트: `12px`

## 4. 카드/섹션 패턴
- 모든 주요 콘텐츠는 라운드 카드 안에 배치한다.
- 카드 내부는 `헤더 / 본문 / 하단 액션 또는 페이징` 구조를 명확히 분리한다.
- 카드 제목이 불필요하게 공간을 차지하면 제거하고, 카드 역할은 배치와 컨트롤 구조로 드러낸다.
- 빈 상태, 비활성 상태, 선택 상태는 시각적으로 분명히 구분한다.

## 5. 상단 검색/액션 바 패턴
- 기본 구성은 `검색 input + 상세필터 버튼 + 필터 상태 표시 + 액션 버튼들`이다.
- 한 줄 배치를 기본으로 한다.
- 검색 input은 너무 짧지 않게 유지하고, 현재 기준 폭은 `280px`이다.
- 액션 버튼은 우측 정렬을 기본으로 한다.
- 주요 액션 버튼 폭은 통일한다.
- 현재 샘플처럼 액션 버튼은 빠른 인지를 위해 이모지 prefix를 유지한다.
- 필터 적용 중이면 `필터 적용됨` 같은 배지와 `초기화` 액션을 노출한다.

## 6. 상세필터 패턴
- 상세필터는 전체 모달이 아니라 버튼 아래 popover를 기본으로 사용한다.
- 필터 항목은 검색 가능한 멀티선택 드롭다운으로 제공한다.
- 선택된 값은 `flat chip` 형태로 표시한다.
- chip의 `x`를 누르면 해당 선택값이 제거되어야 한다.
- 드롭다운에서 체크해도 즉시 목록에 반영하지 않는다.
- `적용 / 취소` 버튼을 눌러야 실제 필터가 반영되도록 한다.
- 기본 placeholder는 `전체`를 사용한다.
- 검색 결과가 없을 때는 `검색 결과 없음`을 표시한다.

## 7. 테이블 패턴
- 헤더 배경색을 본문과 분리해 인지성을 높인다.
- 세로줄 없이 구성한다.
- row hover와 row selected 상태를 모두 제공한다.
- 선택된 row는 배경색과 좌측 포인트 바로 표시한다.
- row 선택과 checkbox 선택은 분리한다.
- checkbox는 checkbox 클릭 시에만 변경된다.
- 정렬 가능한 컬럼은 헤더 버튼과 방향 표시를 제공한다.
- 숫자 컬럼은 우측 정렬을 기본으로 한다.
- 페이징을 기본으로 제공한다.
- 건수 0인 항목은 목록에서 숨기는 패턴을 기본 규칙으로 고려한다. 단, 페이지별 비즈니스 규칙에 따라 예외 가능하다.

## 8. 페이지네이션 패턴
- 하단에 `1-5 / 20건` 형태의 캡션을 제공한다.
- `이전 / 페이지 번호 / 다음` 구조를 기본으로 한다.
- 현재 적용된 검색, 필터, 정렬 결과에 대해 동작해야 한다.

## 9. 상태변경 패턴
- 선택된 항목의 요약 정보를 상단에 배치한다.
- 상태 버튼은 밀도 높은 그리드 형태로 배치한다.
- textarea + 글자수 표시를 함께 제공한다.
- 담당자는 검색 가능한 멀티선택 드롭다운으로 제공한다.
- 선택된 담당자는 chip 형태로 표시하고 `x`로 해제할 수 있어야 한다.
- 저장 버튼은 변경사항이 있을 때만 활성화한다.
- 권한에 따라 상태 선택 범위와 버튼 노출을 분리한다.

## 10. 피드백 패턴
- 저장, 삭제, 복사, 이행점검 결과는 toast로 알려준다.
- 삭제, 일괄수정 같은 확인성 액션은 별도 확인 UI를 제공한다.
- empty state 문구를 각 영역에 제공한다.
- disabled 상태는 opacity, cursor, hover 제거 등으로 명확히 표현한다.

## 11. 권한 시연 패턴
- 관리자 / 일반사용자 전환 UI를 제공한다.
- 관리자 전용 버튼과 일반사용자 제한 상태를 시각적으로 구분한다.
- 테스트용 권한 전환 UI는 사이드바 하단에 배치한다.

## 12. 코드 구조 원칙
- UI 상태와 DOM 렌더링, 이벤트 바인딩은 `app.js`에 둔다.
- 목데이터와 테스트용 비즈니스 로직은 `mock-service.js`에 둔다.
- 공통 Skill에서는 이 구조를 기본 권장 패턴으로 명시한다.
- 페이지가 늘어나면 UI 레이어와 mock/business 레이어를 계속 분리하는 방향으로 확장한다.

## 13. 검증 규칙
- 구현 후 실제 브라우저 렌더링 검증을 기본 절차로 포함한다.
- 검증은 `playwright` skill을 우선 사용한다.
- 검증 기준은 FHD, 브라우저 100% 확대 비율을 기본으로 한다.
- 최소 확인 항목은 다음을 포함한다.
  - 콘솔 에러 유무
  - 초기 진입 시 hidden 처리 요소가 화면을 가리지 않는지
  - 레이아웃 겹침, 줄바꿈 깨짐, overflow 여부
  - 헤더가 현재 샘플과 다른 스타일로 과도하게 단순화되거나 변형되지 않았는지
  - 액션 버튼의 이모지 prefix가 빠지지 않았는지
  - 테이블 글자 크기가 기준보다 작아지지 않았는지
  - 검색, 상세필터, 정렬, 페이징, 모달/팝오버, 드롭다운 동작
  - 권한 전환에 따른 버튼 노출과 상태 제한
  - 클릭, hover, selected, disabled 상태 표현
- 업체 전달용 산출물이나 시각 확인이 필요한 경우에만 `screenshot` skill 또는 캡처를 선택적으로 사용한다.
- 공통 Skill에는 "구현 후 브라우저 검증까지 완료"를 기본 완료 조건으로 넣는다.

## 17. Skill 작성 전 체크리스트
- breadcrumb를 모든 페이지 공통 패턴으로 사용할지 확정한다.
- 폰트 크기, 굵기, 버튼 높이, 폭, spacing 같은 숫자 기준을 공통 규칙으로 확정한다.
- 헤더 스타일을 단순 설명이 아니라 현재 샘플과 유사한 시각 규칙으로 명시했는지 확인한다.
- 주요 액션 버튼의 이모지 포함 여부를 Skill 규칙에 명시했는지 확인한다.
- 테이블 폰트 크기를 하한선처럼 다뤄 임의 축소를 막고 있는지 확인한다.
- 검색/필터/정렬/페이징 패턴을 모든 페이지에 동일하게 적용할지 확정한다.
- 브라우저 검증을 `playwright` 기반 기본 절차로 포함할지 확정한다.
- 페이지 전용 비즈니스 규칙이 공통 Skill로 섞이지 않도록 분리한다.
- 현재 구현과 문서 기준이 어긋나지 않는지 확인한다.
